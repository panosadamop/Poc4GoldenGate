## 1. Context & Goals

eEFKA is migrating a large-scale Oracle-based legacy system to a modern application platform. During the transition period — which may span 12–24 months — both the legacy system and the new system must remain operational and data-consistent. Neither system can afford downtime, and citizen-facing operations (insurance contributions, pension calculations, employer registrations) must continue uninterrupted.

### 1.1 Migration Approach

The chosen pattern is the **Strangler Fig**: the new application gradually takes ownership of functional domains one at a time. Until a domain is fully owned by the new system, the legacy system remains the system of record for that domain.

```
Phase 1: Legacy is authoritative → CDC flows Legacy → New  (read-only new app)
Phase 2: Domain handover         → CDC flows bidirectionally (conflict rules apply)
Phase 3: Legacy decommissioned   → New app is authoritative, legacy is read-only archive
```

### 1.2 Goals

| Goal | Measurable Target |
|---|---|
| Data freshness | Changes available in new DB within **< 5 seconds** under normal load |
| Table coverage | **≥ 99%** of actively-written tables captured |
| Data fidelity | Zero lost events; at-least-once delivery with deduplication |
| Availability | Sync pipeline uptime **≥ 99.9%** |
| Schema evolution | DDL changes in legacy propagated and reconciled within **1 business day** |

---

## 2. Scope

### 2.1 What Is In Scope

- Continuous CDC of **~3 000 tables** across **~100 schemas** in the legacy Oracle database
- Transformation and routing of change events to the new Oracle database
- Handling of data generated by **stored procedures and packages** (captured as resulting DML)
- Bidirectional sync for domains in Phase 2 (handover period)
- Initial bulk load (snapshot) for all in-scope tables

### 2.2 What Is Out of Scope

- Replication of Oracle stored procedure or package **source code** — procedures are re-implemented as application-layer services in the new platform
- Replication of Oracle **sequences** — the new system uses its own identity strategy
- Replication of **temporary tables**, **materialized view logs**, and pure read-model tables (flagged per schema in the Table Registry)
- Real-time DDL propagation — schema changes are governed by a change-management process

---

## 3. Architectural Principles

| Principle | Rationale |
|---|---|
| **Legacy DB is not modified** | No triggers, no additional columns, no application changes on the legacy side |
| **LogMiner-only** | Does not require the XStream add-on license; reads redo logs out-of-band |
| **Domain-partitioned connectors** | One Debezium connector per functional domain, not one global connector |
| **Schema Registry is the contract** | All topic schemas are versioned; consumers never rely on raw SQL definitions |
| **Dead-letter queues everywhere** | Every pipeline stage has a DLQ; no event is silently dropped |
| **Idempotent sinks** | The new DB applies changes using MERGE (upsert) keyed on primary key + SCN |
| **Observability first** | Every component emits metrics to a central dashboard before go-live |

## 4. Oracle Version Compatibility

| Attribute | Legacy DB (Source) | New DB (Target) |
|---|---|---|
| **Oracle version** | **12c** (12.1 or 12.2 — confirm with DBA) | **19c** (Long-Term Support release) |
| **Multitenant** | Likely non-CDB — verify (see §4.1) | CDB + PDB recommended |
| **JDBC driver** | `ojdbc8.jar` from Oracle 19c client (backward-compatible) | `ojdbc8.jar` from Oracle 19c client |
| **Debezium support** | Full from Debezium 2.3+ (12.1 minimum) | N/A — 19c is the sink target only |
| **LogMiner strategy** | `redo_log_catalog` recommended (see §4.2) | N/A |
| **Non-CDB status** | Fully supported in 12c | Non-CDB deprecated in 19c — use PDB |

### 4.1 CDB vs Non-CDB on Oracle 12c

This is the most critical configuration decision for the Debezium connector. Oracle 12c introduced the Multitenant (CDB/PDB) architecture but made it optional — most legacy 12c databases were deployed as traditional non-CDB instances. The two topologies require entirely different connector configurations.

**Verify which topology the legacy DB uses:**

```sql
-- Run as SYSDBA on the legacy 12c instance
SELECT CDB FROM V$DATABASE;
-- YES → Multitenant (CDB)  |  NO → Traditional non-CDB (most common for legacy)
```

| | Non-CDB (most common for legacy 12c) | CDB (12c Multitenant) |
|---|---|---|
| `database.pdb.name` | **Not set** | Set to PDB name |
| LogMiner user prefix | `dbzuser` (no `c##`) | `c##dbzuser` |
| `CONTAINER = ALL` grants | Not required | Required |
| `table.include.list` format | `SCHEMA.TABLE` | `PDB_NAME.SCHEMA.TABLE` |
| Setup script | `oracle/01_cdc_setup_12c_noncdb.sql` | `oracle/01_cdc_setup.sql` |

### 4.2 LogMiner Strategy on Oracle 12c

| Strategy | Recommendation | Reason |
|---|---|---|
| `online_catalog` | **Avoid on 12c** | Reads the current in-memory dictionary; unreliable for DDL tracking on 12.1; can return incorrect column metadata after schema changes |
| `redo_log_catalog` | **Use this** | Mines the data dictionary embedded in the redo log itself; correct for high-table-count environments and schema evolution |

Every domain connector must set:

```json
"log.mining.strategy": "redo_log_catalog"
```

### 4.3 Known Oracle 12c Limitations

| Feature | 12c Status | Mitigation |
|---|---|---|
| **XMLTYPE columns** | Not replicable via LogMiner | Mark tables in Table Registry as `XMLTYPE_EXCLUDED`; replicate via application API or file export |
| **LOB columns** (CLOB, BLOB, NCLOB) | Supported with `lob.enabled=true` + full supplemental logging; known reliability issues on 12.1 | Thorough staging validation per LOB table; treat LOB-heavy tables as a separate connector domain |
| **User-Defined Types (UDTs)** | Not supported by Debezium | Re-implement as flat relational structures in the new 19c schema |
| **INTERVAL data types** | Captured as a string literal | Add type-coercion SMT in the Transformation layer before the 19c sink |
| **Invisible columns** | Not captured by LogMiner | Document in Table Registry; explicitly re-add to 19c schema if required |
| **`LOGMINING` privilege** | Only available from Oracle 12.2 | On 12.1: replace with `EXECUTE ON SYS.DBMS_LOGMNR` + `SELECT ANY DICTIONARY` |
| **LogMiner performance** | Measurably slower than 19c under high DML volume | Set `redo_log_catalog`; tune `log.mining.batch.size.max`; retain archive logs ≥ 72 h |

### 4.4 JDBC Driver Configuration

Use the Oracle **19c JDBC driver** to connect to **both** databases. Oracle JDBC drivers are backward-compatible with older server versions.

```
Kafka Connect (Debezium source) → ojdbc8.jar (19c client) → Oracle 12c  ← backward-compatible ✓
Kafka Connect (JDBC Sink)       → ojdbc8.jar (19c client) → Oracle 19c  ← same version ✓
```

- JDK 8: use `ojdbc8.jar`
- JDK 11+: use `ojdbc11.jar` (recommended — Kafka Connect runs well on JDK 17)
- Download from: Oracle Technology Network → Database 19c JDBC drivers
- Place the JAR at `kafka-connect/ojdbc8.jar` (Dockerfile copies it automatically)

### 4.5 Oracle 19c Advantages for the New Database

| Feature | Benefit |
|---|---|
| **Automatic Indexing** | Oracle auto-creates/drops indexes based on actual query workload — reduces DBA overhead on the new schema as access patterns emerge |
| **Real Application Testing** | Replay captured 12c production workloads against 19c to validate schema equivalence before cutover |
| **Enhanced JSON constraints** | `IS JSON` check constraint on VARCHAR2(32767) — natural fit for event-sourced view tables |
| **Approximate Query Processing** | `APPROX_COUNT_DISTINCT` speeds up reconciliation queries on large tables |
| **PDB isolation** | Each application schema domain can be isolated in its own PDB, simplifying future multi-tenancy |

## 6. Connector Topology

A single Debezium connector mining 3 000 tables creates one LogMiner session spanning all schemas. This creates two problems: (a) the LogMiner session becomes very wide and slow, and (b) a failure in one domain stalls all others.

### 6.1 Domain Grouping Strategy

Connectors are split by **functional domain**, each owning 200–400 tables:

| Connector | Schemas (examples) | Table Count (approx) | Priority |
|---|---|---|---|
| `cdc-insured-persons` | INSURED, PERSONS, DEMOGRAPHICS | ~300 | Critical |
| `cdc-employers` | EMPLOYERS, EMPLOYER_CONTRIB | ~280 | Critical |
| `cdc-contributions` | CONTRIBUTIONS, INSTALLMENTS, DEBTS | ~350 | Critical |
| `cdc-pensions` | PENSIONS, PENSION_CALC, BENEFITS | ~320 | Critical |
| `cdc-health` | HEALTH_INSUR, MEDICAL_CLAIMS | ~260 | High |
| `cdc-accidents` | WORK_ACCIDENTS, OCCUPATIONAL | ~180 | High |
| `cdc-reference-data` | CODES, LOOKUPS, PARAMETERS, TYPES | ~200 | Medium |
| `cdc-audit-legacy` | AUDIT_*, LOG_* | ~150 | Low |
| `cdc-archive` | HIST_*, ARCHIVE_* | ~300 | Low |
| *(additional domains)* | ... | ~460 | — |

> Each connector gets its own Kafka Connect worker task and its own `schema.history.internal` topic.

### 6.2 Connector Configuration Parameters (per-domain)

```json
{

  "log.mining.batch.size.max": "10000",

```

### 6.3 Table Exclusion Rules

Not all 3 000 tables need replication. The **Table Registry** (maintained in Confluence) classifies each table:

| Class | Action | Examples |
|---|---|---|
| `TRANSACTIONAL` | Capture + sync | Core business tables |
| `REFERENCE` | Capture + sync (low frequency) | Code tables, parameter tables |
| `SESSION` | Exclude | `SESSION_*`, `TEMP_*`, `WRK_*` |
| `LEGACY_ONLY` | Exclude | Tables deprecated in new model |
| `MATERIALIZED_VIEW` | Exclude | `MV_*` — rebuilt from events in new DB |
| `QUEUE_TABLE` | Exclude | Oracle AQ internal tables |

## 7. Handling Stored Procedures and Packages

This is the most architecturally significant challenge. Debezium does **not** replicate stored procedure execution — it captures the **resulting DML** (INSERT / UPDATE / DELETE rows) written to the redo log.

### 7.1 What Debezium Captures

When a legacy procedure like `PKG_CONTRIBUTIONS.CALC_MONTHLY_DEBT(p_insured_id)` executes and writes 12 rows to `CONTRIBUTIONS.MONTHLY_DEBT`, Debezium captures all 12 row-level change events. The new database receives the data correctly.

**This covers the majority of cases.**

### 7.2 What Is NOT Captured

| Scenario | Problem | Mitigation |
|---|---|---|
| Procedures that only call other procedures with no DML | No events emitted | Identify and instrument with explicit DML or wrapper |
| Procedures that write to excluded tables (SESSION, TEMP) | Data not replicated | Trace downstream DML; ensure final DML to permanent tables is captured |
| DBMS_OUTPUT / in-memory calculations with no DML | Nothing to capture | Re-implement as application service |
| Autonomous transactions (PRAGMA AUTONOMOUS_TRANSACTION) | Captured, but may appear out of order | SCN-based ordering resolves this |
| DDL executed from PL/SQL (ALTER TABLE inside a procedure) | Schema history may lag | Schema change management process; connector restart policy |
| DBMS_SCHEDULER / DB-level job results | Captured only if the job produces DML | Monitor job completion via separate alerting |

### 7.3 Package & Procedure Inventory

A **procedure audit** must be conducted before go-live:

```
For each package / standalone procedure:
  1. Classify: does it produce DML on in-scope tables?
  2. If yes → verify events flow correctly through CDC pipeline (test in staging)
  3. If no  → flag for re-implementation in new application layer
  4. Document in Confluence under: eEFKA / Migration / Procedure Inventory
```

**Updated estimate: 4–6 person-weeks** using Oracle PL/Scope automated analysis plus manual classification (>1 000 procedures confirmed). See Section 17 Q4 for the tooling approach.

### 7.4 Procedures That Must Be Re-Implemented

Any business logic that is **not** expressible as observable row changes must be migrated to the new application layer. Examples:

- Calculation engines (pension amount, contribution rates) → Java/Python services
- Validation procedures → application-layer validators
- Notification/alert procedures (DBMS_MAIL, UTL_HTTP) → new messaging services
- Scheduled batch recalculations → new batch jobs on the new platform

## 8. Initial Bulk Load (Snapshot)

Before streaming begins, a consistent initial snapshot must load all target tables.

### 8.1 Snapshot Strategy

```
Phase A — Reference data first (codes, lookups, parameters)
          Estimated rows: ~5M — completes in < 1 hour

Phase B — Core transactional domains in parallel
          Estimated rows: ~500M–2B across 3 000 tables
          Estimated duration: 8–24 hours depending on Oracle I/O
          Strategy: connector snapshot.mode = initial
                    snapshot.isolation.mode = read_committed

Phase C — Archive / history tables (low priority, can lag by 24h)
          snapshot.mode = initial_only for these connectors
```

### 8.2 Snapshot Parallelism

Each domain connector runs its snapshot independently. To avoid overwhelming Oracle I/O:

- Run critical-domain snapshots during an agreed **off-peak window** (e.g., 22:00–06:00)
- Stagger connector start times by 15–30 minutes
- Cap `snapshot.fetch.size` at 2 000 rows per batch

### 8.3 Snapshot Validation

After snapshot completes, a **row-count reconciliation job** verifies:

```sql
-- Run on both databases for each table
SELECT COUNT(*) FROM {schema}.{table};
-- Alert if delta > 0.01% (reference data) or > 0 (financial data)
```

## 9. Bidirectional Sync (Phase 2 — Domain Handover)

During the handover period for a given domain, both applications may write to their respective databases. This requires a conflict resolution strategy.

### 9.1 Conflict Sources

| Scenario | Description |
|---|---|
| Concurrent updates | Legacy and new app update the same row within the same second |
| Cascading updates | A procedure on legacy triggers a cascade that arrives out of order |
| Delete vs update | Legacy deletes a row; new app updates the same row before the delete arrives |
| Schema drift | A DDL change on one side is not yet applied to the other |

### 9.2 Conflict Resolution Rules

```
Rule 1 — Last-Write-Wins by SCN
  Each event carries a System Change Number (SCN).
  The higher SCN wins. Applied via MERGE with SCN guard:

  MERGE INTO target t USING source s ON (t.pk = s.pk)
  WHEN MATCHED AND s.source_scn > t.last_sync_scn THEN UPDATE ...
  WHEN NOT MATCHED THEN INSERT ...

Rule 2 — Domain Authority
  During handover, only the authoritative system for a given domain
  writes to the canonical record. The other system writes to a
  shadow/staging table, not the canonical table.

Rule 3 — Conflict Log
  Any event that is rejected by Rule 1 is written to:
    SYNC_METADATA.CONFLICT_LOG (table_name, pk_value, winning_scn,
                                 losing_scn, winning_source, ts)
  This log is reviewed daily by the migration team.

Rule 4 — Financial tables — no bidirectional sync
  Contribution, debt, and pension payment tables are NEVER
  written bidirectionally. A single authoritative source is
  maintained until full cutover.
```

### 9.3 Reverse CDC (New → Legacy)

When the new application is authoritative for a domain, a **reverse Debezium connector** captures changes from the new DB and syncs them back to the legacy DB (for reporting and legacy integrations that cannot be migrated immediately).

The topology is symmetric — the same Debezium + Kafka infrastructure handles both directions.

## 10. Schema Evolution Governance

With 3 000 tables, DDL changes are frequent. The pipeline must not break when a column is added, renamed, or dropped.

### 10.1 Schema Registry Compatibility Mode

All topics use **BACKWARD_TRANSITIVE** compatibility:

- Consumers can always read messages written with any previous schema version
- New nullable columns with defaults are safe to add
- Column renames and drops require a compatibility break process (see below)

### 10.2 DDL Change Process

```
1. Change request filed in Jira (project: EEEFKA-MIGRATION)
2. DBA applies DDL to legacy DB
3. Debezium detects DDL via LogMiner → updates schema.history topic
4. Migration team reviews the schema diff in Schema Registry UI
5. If BACKWARD compatible → auto-deployed to new DB by Liquibase job
6. If BREAKING change    → manual migration script reviewed and approved
                           Connector may need restart with schema.history reset
7. Reconciliation job run after DDL propagation
```

### 10.3 Schema Registry Topic Layout

```
_legacy_schema_history.domain-a      ← per-connector DDL history
_legacy_schema_history.domain-b
...
_schema_registry                     ← Confluent Schema Registry
```

## 11. Data Quality and Reconciliation

### 11.1 Continuous Reconciliation

A lightweight **reconciliation service** runs every 15 minutes against a random sample of 50 tables:

```
For each sampled table:
  1. Pick a random PK range covering ~1 000 rows
  2. Hash the rows on both databases (MD5 of serialised row)
  3. If hashes differ → log to SYNC_METADATA.RECONCILIATION_ISSUES
  4. If divergence > 0.1% → trigger PagerDuty alert (P2)
  5. If divergence > 1%   → trigger PagerDuty alert (P1) + auto-pause connector
```

### 11.2 Full Nightly Reconciliation

For financial tables (contributions, debts, pensions), a **full count + checksum reconciliation** runs nightly:

- Compares row counts and SUM of key financial columns
- Results written to `SYNC_METADATA.NIGHTLY_RECONCILIATION`
- Any discrepancy > 0 for financial tables triggers immediate P1

### 11.3 Dead-Letter Queue Processing

```
DLQ topic: dlq.legacy.{SCHEMA}.{TABLE}

Events land in DLQ when:
  - Sink connector cannot apply the event (constraint violation, type mismatch)
  - Transformation fails (null PK, corrupt Avro)

DLQ processor (daily batch job):
  1. Parse the failed event and root cause
  2. Categorise: schema_mismatch | constraint_violation | data_error
  3. Apply fix (re-derive, skip, or manual correction)
  4. Re-publish to original topic with corrected payload
```

## 12. Security and Compliance (GDPR / Greek DPA)

eEFKA handles sensitive citizen data. The pipeline must comply with GDPR and the requirements of the Hellenic Data Protection Authority (HDPA).

### 12.1 PII Masking in Transit

Certain fields must be masked before they appear in Kafka topics (which may be accessible to more systems than the databases):

| Field Type | Masking Strategy |
|---|---|
| AMKA (social security number) | SHA-256 hash with HMAC salt, reversible only by authorised services |
| Full name | Keep in topic (needed for sync), restricted topic ACL |
| Tax number (AFM) | SHA-256 hash with HMAC salt |
| Bank account (IBAN) | Last 4 digits only in Kafka; full value only in DB-to-DB sync path |
| Medical data | Full field encryption (AES-256), decrypted only at sink |

### 12.2 Topic Access Control

Kafka ACLs:
- CDC capture connectors: WRITE to `legacy.*` topics
- Transformation services: READ from `legacy.*`, WRITE to `transformed.*`
- Sink connectors: READ from `transformed.*`
- Audit service: READ from all topics
- No external system reads raw CDC topics

### 12.3 Data at Rest

- Kafka brokers: encryption at rest (AWS KMS or on-prem equivalent)
- Schema Registry: TLS required
- Oracle LogMiner user: password rotation every 90 days via Vault

### 12.4 Audit Trail

Every event that flows through the pipeline is recorded in a tamper-evident audit log:

```
audit.cdc.events topic:
  { event_id, source_table, operation, pk_value, scn,
    pipeline_ts, connector_id, schema_version }
```

Retained for 7 years (statutory requirement for social security data).

> LogMiner sessions on Oracle require adequate SGA/PGA. Coordinate with the DBA team:
> `LOG_ARCHIVE_DEST_1` should have at least 72 h of redo log retention.

> **Two-DC deployment (primary + DR):** Distribute Kafka brokers across both data centres (3 in primary DC + 2 in DR DC). Kafka Connect workers: 3 in primary, 2 standby in DR. Oracle 19c DR is handled by Oracle Data Guard — the CDC pipeline does not write directly to the DR Oracle instance. Total additional nodes for DR capability: ~4 (2 Kafka brokers + 2 MirrorMaker 2).

## 16. Risks and Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| LogMiner session overwhelmed by 3 000-table scan | High | High | Domain-partitioned connectors; tune `log.mining.batch.size.max` |
| Oracle archive log gap (redo log purged before consumed) | Medium | Critical | Set `LOG_ARCHIVE_DEST_1` retention ≥ 72 h; monitor lag continuously |
| PL/SQL procedure producing DML not captured | Medium | High | Full procedure audit in Phase 0; staging validation for each domain |
| DDL breaking change disrupts connector | Medium | High | Schema change process; connector restart playbook |
| New and legacy apps writing conflicting data | Medium | Critical | Financial tables: single writer enforced; conflict log reviewed daily |
| GDPR breach via unsecured Kafka topic | Low | Critical | Topic ACLs; PII masking SMT; encryption at rest |
| Snapshot duration exceeds maintenance window | Medium | Medium | Stagger connector starts; run snapshots off-peak; use `snapshot.select.statement.overrides` to limit row range |
| Schema Registry becoming a single point of failure | Low | High | Deploy SR in HA mode (3 nodes); topic replication factor ≥ 3 |

## 17. Open Questions

1. **LogMiner vs XStream** — ✅ RESOLVED: eEFKA does **not** hold an Oracle XStream license. LogMiner with `redo_log_catalog` strategy is confirmed as the CDC mechanism for all domains. XStream is not applicable.

2. **Network topology** — ✅ RESOLVED: Two **on-premises data centres** (primary DC + DR DC). Architectural implications:
   - Kafka cluster spans both DCs: **3 brokers in primary DC + 2 brokers in DR DC** (5-node cluster, rack-aware replication with `min.insync.replicas=2`)
   - Debezium connectors run in primary DC, connecting to the primary Oracle 12c instance
   - **Kafka MirrorMaker 2** replicates all CDC topics from primary Kafka to a passive Kafka cluster in the DR DC for failover capability
   - JDBC sink connectors in primary DC write to Oracle 19c (primary); Oracle 19c DR is handled by **Oracle Data Guard** — the CDC pipeline does not replicate directly to the DR Oracle instance
   - See updated infrastructure sizing note in Section 13

3. **Oracle version** — ✅ RESOLVED: Legacy DB is Oracle **12c**, new DB is Oracle **19c**. Full version-specific guidance in Section 4.

4. **Procedure inventory size** — ✅ RESOLVED: **> 1 000 packages and procedures**. The original 2 person-week estimate is insufficient. Updated approach:
   - Use Oracle **PL/Scope** (`ALTER SESSION SET PLSCOPE_SETTINGS='IDENTIFIERS:ALL'`) to automatically extract procedure call graphs and DML footprint
   - Supplement with `ALL_SOURCE`, `ALL_DEPENDENCIES`, and `ALL_OBJECTS` queries for full coverage
   - Estimated effort revised to **4–6 person-weeks** with one dedicated analyst
   - Phase 0 updated to include PL/Scope tooling setup and automated inventory generation before manual classification

5. **Bidirectional domains** — ✅ RESOLVED (deferred): No domains are pre-designated as bidirectional. Sync direction will be assessed **per domain during Phase 4** handover planning based on migration readiness at that time. Constraints that apply regardless:
   - Financial tables (contributions, debts, pensions) remain **strictly one-way** (legacy → new) until full cutover, with no exceptions
   - Any domain proposed for bidirectional sync must pass a conflict-resolution design review before Phase 4 activation

6. **Compliance scope (PII data map)** — ⚠️ PARTIALLY RESOLVED: DPO audit is **in progress**. Known PII-bearing schemas based on current analysis:

   | Schema | PII Fields Present |
   |---|---|
   | INSURED / PERSONS | AMKA, full name, date of birth, residential address |
   | HEALTH_INSUR / MEDICAL_CLAIMS | Medical condition codes, diagnosis data, provider details |
   | CONTRIBUTIONS / DEBTS | IBAN (partial capture), AFM, employer tax registration |
   | EMPLOYERS | AFM, company registration number, authorised representative data |

   The PII masking SMT **must be configured and validated before any domain's topics go live in production**. Full PII data map is a **Phase 0 gate criterion** — no connector is registered in production until all relevant tables are confirmed in the Table Registry with their masking rules documented.
