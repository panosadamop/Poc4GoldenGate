1. Context & Goals

eEFKA is migrating a large-scale Oracle-based legacy system to a modern application platform. During the transition period — which may span 12–24 months — both the legacy system and the new system must remain operational and data-consistent. Neither system can afford downtime, and citizen-facing operations (insurance contributions, pension calculations, employer registrations) must continue uninterrupted.

2. Scope

2.1 In Scope

  • Continuous CDC of ~3 000 tables across ~100 schemas in the legacy Oracle database
  • Transformation and routing of change events to the new Oracle database
  • Data produced by stored procedures and packages (captured as resulting DML — see Section 6)
  • Bidirectional sync for domains in Phase 2 handover period
  • Initial bulk load (snapshot) for all in-scope tables

2.2 Out of Scope

  • Replication of Oracle stored procedure or package source code — procedures are re-implemented as application-layer services
  • Replication of Oracle sequences — the new system uses its own identity strategy
  • Temporary tables, materialized view logs, and pure read-model tables (flagged in the Table Registry)
  • Real-time DDL propagation — schema changes are governed by a change-management process (Section 9)

3. Architectural Principles

PrincipleRationale
Legacy DB is not modified No triggers, no additional columns, no application changes on the legacy side. LogMiner reads redo logs out-of-band.
LogMiner-only (no XStream) Does not require the Oracle XStream add-on license. Evaluate XStream for financial domains if license is available.
Domain-partitioned connectors One Debezium connector per functional domain, not one global connector. Isolates failures and enables independent scaling.
Schema Registry is the contract All topic schemas are versioned in Confluent Schema Registry. Consumers never rely on raw Oracle column definitions.
Dead-letter queues everywhere Every pipeline stage has a DLQ. No event is silently dropped.
Idempotent sinks New DB applies changes using MERGE (upsert) keyed on primary key + SCN guard. Safe to replay any event.
Observability first Every component emits metrics to Grafana before go-live. No domain is promoted without a working dashboard.

6. Connector Topology

Why not one connector for all 3 000 tables?

A single Debezium connector mining 3 000 tables creates one LogMiner session spanning all schemas. This causes two problems: (a) the LogMiner session becomes very wide and slow, and (b) a failure or lag spike in one domain stalls all others. Domain-partitioned connectors solve both issues.

7. Handling Stored Procedures and Packages

What Debezium captures

Debezium does not replicate stored procedure execution — it captures the resulting DML (INSERT / UPDATE / DELETE rows) written to the redo log. When a legacy procedure such as PKG_CONTRIBUTIONS.CALC_MONTHLY_DEBT(p_insured_id) writes 12 rows to CONTRIBUTIONS.MONTHLY_DEBT, Debezium captures all 12 row-level change events. The new database receives the data correctly. This covers the majority of cases.

7.1 Scenarios NOT Automatically Captured

ScenarioProblemMitigation
Procedures that only call other procedures with no DML No events emitted Identify and instrument with explicit DML or wrapper
Procedures writing to excluded tables (SESSION, TEMP) Data not replicated Trace downstream DML; ensure final write to permanent tables is captured
DBMS_OUTPUT / in-memory calculations with no DML Nothing to capture Re-implement as application service
PRAGMA AUTONOMOUS_TRANSACTION May appear out of order SCN-based ordering at the sink resolves this
DDL executed from PL/SQL (ALTER TABLE inside a procedure) Schema history may lag Schema change management process; connector restart policy
DBMS_SCHEDULER / DB-level job results Captured only if the job produces DML Monitor job completion via separate alerting

9. Bidirectional Sync (Phase 2 — Domain Handover)

Financial tables are NEVER written bidirectionally

Contribution, debt, and pension payment tables must have exactly one authoritative writer at any point in time. A conflict in these tables is a regulatory problem, not just a data integrity problem. A single writer is enforced until full cutover for all financial domains.

9.1 Conflict Sources

ScenarioDescription
Concurrent updatesLegacy and new app update the same row within the same second
Cascading updatesA procedure on legacy triggers a cascade that arrives out of order
Delete vs updateLegacy deletes a row; new app updates the same row before the delete arrives
Schema driftA DDL change on one side is not yet applied to the other

9.2 Conflict Resolution Rules

9.3 Reverse CDC (New → Legacy)

When the new application becomes authoritative for a domain, a reverse Debezium connector captures changes from the new DB and syncs them back to the legacy DB, supporting legacy integrations that cannot be migrated immediately. The topology is symmetric — the same Debezium + Kafka infrastructure handles both directions.

11.2 Full Nightly Reconciliation (financial tables)

For financial tables (contributions, debts, pensions) a full count + checksum reconciliation runs nightly:

  • Compares row counts and SUM of key financial columns
  • Results written to SYNC_METADATA.NIGHTLY_RECONCILIATION
  • Any discrepancy > 0 on financial tables triggers immediate P1

11.3 Dead-Letter Queue Processing

DLQ processing details

DLQ topic pattern: dlq.legacy.{SCHEMA}.{TABLE}

Events land in DLQ when:

  • Sink connector cannot apply the event (constraint violation, type mismatch)
  • Transformation fails (null PK, corrupt Avro payload)

DLQ processor (daily batch job):

  1. Parse the failed event and root cause
  2. Categorise: schema_mismatch | constraint_violation | data_error
  3. Apply fix (re-derive, skip with justification, or manual correction)
  4. Re-publish to original topic with corrected payload

 Two-DC deployment (primary + DR)

With a primary + DR on-premises deployment, distribute the Kafka cluster across both data centres:

  • Primary DC: 3 Kafka brokers + 3 Kafka Connect workers + Schema Registry primary
  • DR DC: 2 Kafka brokers + 2 standby Kafka Connect workers + Schema Registry replica
  • Add Kafka MirrorMaker 2 (1 process per DC) for cross-DC topic replication
  • Oracle 19c DR is handled by Oracle Data Guard — the CDC pipeline does not write directly to the DR Oracle instance
  • Total additional nodes for DR capability: ~4 (2 Kafka brokers + 2 MirrorMaker 2)

17. Open Questions

  1. GreenRESOLVED LogMiner vs XStream
    eEFKA does not hold an Oracle XStream license. LogMiner with redo_log_catalog strategy is confirmed as the CDC mechanism for all domains. XStream is not applicable.

  2. GreenRESOLVED Network topology
    Two on-premises data centres (primary DC + DR DC). Architectural implications:

    • Kafka cluster spans both DCs: 3 brokers in primary DC + 2 brokers in DR DC (5-node cluster, rack-aware replication, min.insync.replicas=2)
    • Debezium connectors run in primary DC against the primary Oracle 12c instance
    • Kafka MirrorMaker 2 replicates all CDC topics from primary Kafka to a passive DR Kafka cluster for failover capability
    • JDBC sink connectors write to Oracle 19c primary; Oracle 19c DR is handled by Oracle Data Guard — the CDC pipeline does not replicate directly to the DR Oracle instance
    • See the two-DC note in Section 13 for updated infrastructure sizing

  3. GreenRESOLVED Oracle version of legacy DB
    Legacy DB is Oracle 12c, new DB is Oracle 19c. Full version-specific guidance in Section 4.

  4. GreenRESOLVED Procedure inventory size
    > 1 000 packages and procedures confirmed. The original 2 person-week estimate is insufficient. Updated approach:

    • Use Oracle PL/Scope (ALTER SESSION SET PLSCOPE_SETTINGS='IDENTIFIERS:ALL') to automatically extract procedure call graphs and DML footprint
    • Supplement with ALL_SOURCE, ALL_DEPENDENCIES, and ALL_OBJECTS queries for full coverage
    • Revised effort: 4–6 person-weeks with one dedicated analyst
    • Phase 0 updated to include PL/Scope tooling setup before manual classification begins

  5. YellowDEFERRED Bidirectional domains
    No domains are pre-designated as bidirectional. Sync direction will be assessed per domain during Phase 4 handover planning based on migration readiness at that time. Constraints that apply regardless:

    • Financial tables (contributions, debts, pensions) remain strictly one-way (legacy → new) until full cutover — no exceptions
    • Any domain proposed for bidirectional sync must pass a conflict-resolution design review before Phase 4 activation
